# 关系判定方法参考

本文件是 `knowledge-cluster-builder` 的判定细则，供构建 `prerequisites`、`related` 和 `role` 时参考。核心目标：产出**干净、无悬空、无环、可拓扑排序**的依赖图，避免参考仓库半成品里出现过的两类缺陷——前置写成中文标题（悬空）与依赖成环。

## 一、依赖（prerequisites） vs 关联（related）

对任意一对点 A、B，用一句话测试判定：

| 提问 | 结论 |
|---|---|
| “不先掌握 B，是否根本无法开始学 A？” → 是 | A 的 `prerequisites` 含 B（有向、必需） |
| “A、B 相关/易混淆/常对比，但先学哪个都行” | 互放入 `related`（无向、可选） |
| “它们只是碰巧同簇” | **不建立任何关系** |

要点：

- `prerequisites` 是**学习必需的先修**，不是“主题相关”。同簇、同 `kind`、共享关键词都不足以构成依赖。
- 依赖**可以跨簇**（如“反向传播”依赖另一簇的“求导”）。
- 只记**直接前置**。若 A 依赖 B、B 依赖 C，则 A 不必再列 C——间接依赖靠拓扑传递。多列间接祖先会让图变脏、也更易成环。
- `prerequisites` 与 `related` **互斥**：已是前置关系的一对点，不再放进 `related`。
- 宁可少连、不可错连。`scopeStatus` 为 `boundary`/`needs-review` 的点尤其保守。

## 二、用 kind 作依赖先验

上游的 `kind` 是很强的方向信号。常见的“通常依赖”方向（→ 表示“依赖”）：

| 点的 kind | 通常依赖 | 说明 |
|---|---|---|
| `algorithm` | `model` / `method` / `concept` | 算法建立在其形式模型和基础概念上 |
| `method` | `concept` / `model` | 方法建立在它操作的概念/模型上 |
| `model` | `concept` | 模型是对基础概念的形式化 |
| `theorem` | `concept` / `model` | 定理陈述涉及的概念先要成立 |
| `metric` | `task` / `model` | 度量依赖被评价的任务或模型 |
| `task` | `concept` | 任务定义依赖其核心概念 |
| `phenomenon` | `concept` / `model` | 现象需要用其机制概念来解释 |
| `concept` | （通常是被依赖方） | 基础概念很少依赖 algorithm/metric |

这是先验不是铁律：以“学习必需”的实质判断为准，`kind` 只用来提示可能的方向、并帮助避免把方向搞反（成环的常见来源）。

## 三、层级角色 role 判定

先把 `prerequisites` 图建好，再对**每个簇内部**统计两个量：

- **簇内入度** = 该点被同簇其他点依赖的次数（“有多少点以它为前置”）。
- **簇内出度** = 该点依赖的同簇其他点的次数。

据此定角色：

- `trunk`（主干）：簇内出度低（≈0，几乎不依赖簇内其他点）且入度高（被多点依赖），或 `importance` 高、`difficulty` 偏低的枢纽概念。**每个非空簇至少一个 trunk。**
- `leaf`（叶子）：簇内入度为 0（不被任何点依赖）且出度 > 0 的末端点。
- `branch`（分支）：其余介于两者之间的点。

一致性自查：标为 `leaf` 的点不应出现在任何点的 `prerequisites` 里；标为 `trunk` 的点通常应被至少一个点依赖（单点簇或纯并列簇例外）。

## 四、环的产生与打破

**为什么会成环**：把双向“相关”误当双向“依赖”（应放 `related`）；或把依赖方向搞反（B 其实是 A 的前置，却写成 A 是 B 的前置）；或过度添加间接依赖。先按第一、二节收紧判定，能消除大部分环。

**如何打破**：若仍检出环，逐条反向边处理，保留“更基础 → 更进阶”的方向：

1. 取环上一条边 `X → Y`（X 依赖 Y）与其反向 `Y → X`；
2. 判定谁更基础：`importance` 更高者更基础；`importance` 接近（差 < 0.1）时，`difficulty` 更低者更基础；
3. 保留“较进阶点依赖更基础点”的边，移除相反方向那条；
4. 在 `generation.brokenCycleEdges` 追加 `{ from, to, reason }`（reason 说明依据，如 `"importance 0.9>0.6，保留 linked-list→array"`）；
5. 重新检测，直到无环。

无法确信方向时，优先把这对关系降级为 `related`（无向），而不是强行保留一条可能错误的依赖。

## 五、id 纪律（硬约束）

- `prerequisites` / `related` 的每个元素都必须是候选池中**存在的点 id**（kebab-case）。
- 禁止写中文标题或别名（这是参考仓库 18 条悬空引用的根因）。
- 禁止自环（点不依赖/关联自己）。
- 数组内去重。

以上任一违反都会被 `scripts/check-dag.mjs` 拦下并以非零码退出。

# 候选知识点生成

## 目标与边界

输入一个课程名或领域名，输出扁平、去重、适合后续聚类的候选知识点池。

本阶段只回答“哪些知识点应进入候选集”，不回答“它们如何分组或连接”：

- 不创建知识簇、分类树或章节归属。
- 不生成 `clusterId`、`prerequisites`、`related_points` 或其他关系。
- 不生成 `pos`、`scale`、`polygon`、`labelPos` 等布局字段。
- 不扩写 `coreIdea`、`principles`、`applications` 等完整详情。

候选池是 CKDS 发布数据之前的中间产物，不可直接当作 `index.json` 或 `points/*.json` 发布。被接受的 `id`、`title`、`shortSummary`、`difficulty`、`importance` 和 `keyTerms` 可在后续阶段复用。

## 输入解释

唯一必填输入是课程名或领域名。用户提供的受众、深度、语言、范围和排除项优先于默认值。

只有名称时：

- 课程名：按该课程通常对应的一学期主流教学范围解释，并采用其惯常受众层次。
- 领域名：覆盖主流基础到中级知识，排除前沿微主题、品牌产品和短期热点。
- 输出语言：`zh-CN`；保留常用英文名、缩写和别称。

先判断输入属于 `course` 还是 `domain`，并在结果中明确受众、深度、纳入范围和排除范围。若同一名称存在会显著改变候选集的多种解释，只询问一个能消除最大歧义的问题；否则采用最常见解释并显式记录假设。

## 工作流

### 1. 建立范围契约

用一句话分别写清：

1. 目标学习者；
2. 覆盖深度；
3. 纳入边界；
4. 排除边界。

不要用固定数量代替范围定义。课程或领域规模不同，候选数由覆盖度和粒度决定。

### 2. 收集可追溯依据

外部检索可用时，至少使用 3 个相互独立的权威来源，并覆盖至少 2 类来源：

1. 官方课程标准、培养方案或专业知识体系；
2. 公认教材的目录、术语表或索引；
3. 高质量大学课程大纲；
4. 专业组织标准、权威手册或领域综述。

打开并核对原始页面或文档；搜索摘要不能作为唯一依据。每个来源记录 `id`、类型、标题、定位信息和访问日期。不得编造来源、URL、ISBN 或来源支持关系。

无法外部检索时，将 `evidenceMode` 设为 `model-only`，来源数组保持为空，候选项的 `sourceRefs` 保持为空、`confidence` 不得高于 `0.6`，并把边界项放入复核队列。

### 3. 高召回提取

从来源的目录、学习目标、术语表和核心正文中提取所有可能独立教学和考核的概念。可在内部按来源章节或覆盖维度扫描，但这些分组只用于查漏，不得输出为知识簇或候选项归属。

大领域分批提取，每批 50–100 项；每批后与全量候选池去重。所有范围维度均已覆盖，且连续两轮独立查漏各自新增的有效候选不足现有总数的 5% 时，才视为基本饱和。5% 是停止检查阈值，不是删除长尾知识的依据。

### 4. 执行知识点原子化

候选项必须是可独立定义、讲解并设计至少一个考核问题的教学单元。允许的 `kind`：

- `concept`：概念、对象或性质；
- `method`：通用方法或流程；
- `theorem`：定理、定律、原则或关键结论；
- `model`：形式模型或理论框架；
- `algorithm`：有明确输入、过程和输出的算法；
- `task`：领域内稳定、可学习的任务；
- `metric`：有独立解释价值的度量或评价方法；
- `phenomenon`：需要独立解释的现象或效应。

处理规则：

- 并列项能分别定义和考核时拆分；固定术语或不可分机制不拆分。
- 章节名、宽泛分类和“……概述”通常不是知识点；若其本身是公认且可独立教学的概念，可保留。
- 具体产品、库、数据集、人物、机构和单一案例通常作为例子，不作为知识点；只有在主流课程中被稳定独立教授时才保留。
- 学习目标、教学活动、作业名称和行政信息一律排除。
- 上位概念与下位知识点可以同时存在，前提是二者都具有独立教学价值；不要把“相关”误判为“重复”。

### 5. 规范化与去重

1. 使用最常见、最准确的中文术语作为 `title`。
2. 将英文名、缩写和真实同义词放入 `aliases`。
3. 只合并指向同一概念的同义词；近义概念、上下位概念和容易混淆的概念必须保留为不同候选项。
4. 同名异义项在标题中加入最短必要限定词。
5. `id` 使用小写 ASCII kebab-case，优先采用公认英文术语；同一输入与同一规范化结果应得到同一 `id`。
6. `id` 冲突时使用有语义的限定词解决，禁止使用随机编号。

### 6. 补全候选元数据

每个候选项必须包含：

- `id`：课程内唯一，匹配 `^[a-z0-9]+(?:-[a-z0-9]+)*$`；
- `title`：规范中文名；
- `aliases`：去重后的英文名、缩写或别称，可为空；
- `kind`：原子化类型；
- `shortSummary`：30–100 个中文字符，不重复标题，说明“它是什么或解决什么问题”；
- `difficulty`：相对目标学习者的 `基础`、`中等` 或 `进阶`；
- `importance`：相对当前范围的 `[0, 1]` 分值，最多保留两位小数；
- `keyTerms`：2–8 个去重关键词；
- `sourceRefs`：支持该候选项的来源 ID；
- `confidence`：候选项身份、范围归属和命名正确性的 `[0, 1]` 置信度；
- `scopeStatus`：`core`、`boundary` 或 `needs-review`。

评分使用可解释档位，避免无依据的伪精确：

- `importance`：`0.90–1.00` 为不可缺少的核心，`0.70–0.89` 为主干，`0.40–0.69` 为支撑，低于 `0.40` 为边界项。
- `confidence`：`0.90–1.00` 需多个独立权威来源且术语无歧义；`0.70–0.89` 需一个权威来源或多个可靠次级来源；`0.50–0.69` 表示依据有限；低于 `0.50` 必须进入复核队列。

来源出现频次只能作为重要性证据之一，不能把“出现次数”直接等同于教学价值。

### 7. 全局复核

依次执行：

1. **范围复核**：删除明显越界项，把合理但有争议的项标为 `boundary` 或 `needs-review`。
2. **粒度复核**：检查过宽章节和过细实现细节。
3. **同义词复核**：检查中英文别名、缩写、旧称和翻译差异导致的重复。
4. **可区分性复核**：任意两个候选项应能用一句话说明差异。
5. **覆盖复核**：仅用覆盖维度查漏，不输出预设分类结果。
6. **证据复核**：每个 `sourceRefs` 必须存在；无证据项不得伪装成已验证项。

## 输出契约

默认写出单个 UTF-8 标准 JSON 文件 `candidate-points.json`，不含注释和尾随逗号，并按 [candidate-points.schema.json](candidate-points.schema.json) 校验。

```json
{
  "schema_version": "candidate-points/1.0",
  "subject": {
    "id": "data-structures",
    "input": "数据结构",
    "normalizedTitle": "数据结构",
    "inputType": "course",
    "language": "zh-CN",
    "audience": "计算机相关专业本科生",
    "depth": "一学期核心课程",
    "scope": "覆盖数据结构的表示、操作、复杂度与典型算法。",
    "exclusions": ["特定编程语言容器库的 API"]
  },
  "generation": {
    "evidenceMode": "researched",
    "generatedAt": "2026-01-01",
    "candidateCount": 1
  },
  "sources": [
    {
      "id": "src-01",
      "type": "textbook",
      "title": "教材来源标题",
      "locator": "ISBN 或可复核的章节定位",
      "accessedAt": "2026-01-01"
    },
    {
      "id": "src-02",
      "type": "university-course",
      "title": "大学课程大纲标题",
      "locator": "课程大纲 URL",
      "accessedAt": "2026-01-01"
    },
    {
      "id": "src-03",
      "type": "official-standard",
      "title": "课程或专业标准标题",
      "locator": "标准文档 URL",
      "accessedAt": "2026-01-01"
    }
  ],
  "candidates": [
    {
      "id": "linked-list",
      "title": "链表",
      "aliases": ["Linked List"],
      "kind": "concept",
      "shortSummary": "由多个节点及其链接关系构成、支持动态插入和删除的线性数据结构。",
      "difficulty": "基础",
      "importance": 0.85,
      "keyTerms": ["节点", "链接", "线性结构"],
      "sourceRefs": ["src-01", "src-02", "src-03"],
      "confidence": 0.9,
      "scopeStatus": "core"
    }
  ],
  "reviewQueue": []
}
```

`reviewQueue` 中每项必须说明待复核术语、问题类型、原因和建议动作。问题类型仅允许：

- `scope-ambiguity`
- `granularity`
- `synonym`
- `naming`
- `insufficient-evidence`

最终输出保持扁平且可确定性比较：`sources` 按 `id` 排序，`candidates` 按 `id` 排序，`reviewQueue` 按 `term` 排序。除 JSON 外，只报告候选总数、核心/边界/待复核数量、来源数量和仍需人工决定的问题。

## 发布前检查

- [ ] JSON 通过配套 Schema 校验。
- [ ] `candidateCount` 等于 `candidates` 实际长度。
- [ ] `researched` 模式至少有 3 个独立来源并覆盖至少 2 种来源类型。
- [ ] 所有 `id` 唯一、稳定并符合 kebab-case。
- [ ] 所有 `sourceRefs` 都能引用到真实来源。
- [ ] 标题、别名和关键词中不存在明显同义重复。
- [ ] 每项均满足独立定义、独立讲解和独立考核测试。
- [ ] 没有聚类、关系、布局或完整详情字段越界进入本阶段。
- [ ] 低置信度、边界和命名歧义均进入复核队列。
- [ ] 输出不是伪装成 CKDS 发布文件的中间数据。
